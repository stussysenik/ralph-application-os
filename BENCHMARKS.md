# Benchmarks

## Purpose

Benchmarks define what "works" before the platform is allowed to claim generality.

## First Benchmark Families

1. Ramp-like approvals and spend controls
2. Linear-like issue and workflow tracking
3. Notion-like structured workspace and knowledge flows
4. Screenshot-studio capture, annotation, and sharing flows

## First Evaluation Axes

- semantic correctness
- workflow fidelity
- permission fidelity
- migration safety
- edit survival
- artifact complexity

## Rule

No new architectural claim is accepted without either:

- a benchmark improvement, or
- a reduction in complexity with benchmark parity

## Demonstration Goal

The first demo should be able to:

- print the semantic world model for each benchmark family
- build an application blueprint from that model
- run invariant proofs over the benchmark
- explain what kinds of applications the platform can already reason about
- execute a tracked job file and persist reportable Ralph loop artifacts
